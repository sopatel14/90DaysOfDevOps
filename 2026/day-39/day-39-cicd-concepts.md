# Day 39 – What is CI/CD?

## Objective

Today's goal was to understand the concepts behind CI/CD before building actual pipelines.

CI/CD helps development teams automate software delivery, reduce human errors, and release applications faster and more reliably.

---

# Task 1: The Problem

## Scenario

Imagine a team of 5 developers working on the same application and manually deploying changes to production.

### What Can Go Wrong?

* Developers may overwrite each other's changes.
* Bugs may reach production without testing.
* Different environments may behave differently.
* Manual deployment steps can be forgotten.
* Rollbacks become difficult when issues occur.
* Deployments take longer and become stressful.

### What Does "It Works on My Machine" Mean?

This phrase means an application works correctly on a developer's local machine but fails in another environment such as testing, staging, or production.

Common reasons:

* Different operating systems
* Different software versions
* Missing dependencies
* Different environment variables
* Configuration mismatches

This is a real problem because users run the application in environments different from the developer's laptop.

### How Many Times a Day Can a Team Safely Deploy Manually?

Manual deployments are risky and time-consuming.

Most teams can safely deploy only a few times per day manually, whereas automated CI/CD pipelines allow teams to deploy many times per day with confidence.

# Task 2: CI vs CD

## Continuous Integration (CI)

Continuous Integration is the practice of frequently merging code changes into a shared repository.

Every commit triggers automated builds and tests to detect issues early before they reach production.

### Real-World Example

A developer pushes code to GitHub. GitHub Actions automatically builds the application and runs unit tests.

---

## Continuous Delivery (CD)

Continuous Delivery extends CI by automatically preparing software for release after successful testing.

The application is always in a deployable state, but production deployment requires manual approval.

### Real-World Example

After passing tests, an application is automatically deployed to a staging environment and waits for approval before production release.

---

## Continuous Deployment (CD)

Continuous Deployment goes one step further by automatically deploying every successful change to production without human intervention.

Only fully tested and verified changes are released automatically.

### Real-World Example

Companies such as Netflix deploy changes automatically after all tests and quality checks pass.

---

# CI vs Continuous Delivery vs Continuous Deployment

| Feature                         | Continuous Integration | Continuous Delivery | Continuous Deployment |
| ------------------------------- | ---------------------- | ------------------- | --------------------- |
| Automated Build                 | Yes                    | Yes                 | Yes                   |
| Automated Testing               | Yes                    | Yes                 | Yes                   |
| Deployment to Staging           | Optional               | Yes                 | Yes                   |
| Manual Approval Required        | N/A                    | Yes                 | No                    |
| Automatic Production Deployment | No                     | No                  | Yes                   |

# Task 3: Pipeline Anatomy

## Trigger

An event that starts a pipeline.

Examples:

* Git push
* Pull request
* Scheduled execution
* Manual trigger

---

## Stage

A logical phase of the pipeline.

Examples:

* Build
* Test
* Deploy

---

## Job

A collection of related tasks executed within a stage.

Example:

A Build stage may contain a job that compiles source code.

---

## Step

A single action or command inside a job.

Examples:

```bash
npm install
npm test
docker build .
```

---

## Runner

The machine that executes pipeline jobs.

Examples:

* GitHub-hosted runner
* Self-hosted runner
* Jenkins agent

---

## Artifact

Files generated by a pipeline and passed between stages.

Examples:

* Docker images
* Build packages
* JAR files
* Test reports

# Task 4: Pipeline Diagram

## CI/CD Pipeline Example

Scenario:

A developer pushes code to GitHub. The application is tested, built into a Docker image, and deployed to a staging server.

```text
Developer Pushes Code
          │
          ▼
 ┌─────────────────┐
 │     Build       │
 │ Install Deps    │
 │ Compile App     │
 └─────────────────┘
          │
          ▼
 ┌─────────────────┐
 │      Test       │
 │ Unit Tests      │
 │ Validation      │
 └─────────────────┘
          │
          ▼
 ┌─────────────────┐
 │ Docker Build    │
 │ Create Image    │
 │ Push Artifact   │
 └─────────────────┘
          │
          ▼
 ┌─────────────────┐
 │ Deploy Staging  │
 │ Update Server   │
 │ Verify Health   │
 └─────────────────┘
```

### Stages Used

1. Build
2. Test
3. Docker Build
4. Deploy

# Task 5: Explore in the Wild

## Repository Chosen

React

Repository:

https://github.com/facebook/react

### Workflow Examined

`.github/workflows/runtime_build_and_test.yml`

### What Triggers It?

The workflow runs when:

* Code is pushed
* Pull requests are created or updated

### How Many Jobs Does It Have?

The workflow contains multiple jobs responsible for different testing and validation tasks.

### What Does It Do?

Based on the workflow structure, it:

* Installs dependencies
* Builds React packages
* Runs automated tests
* Verifies code quality
* Ensures changes do not break existing functionality

This helps maintain stability in a large open-source project with thousands of contributors.

# Key Learnings

### 1. CI/CD Is a Practice, Not a Tool

GitHub Actions, Jenkins, GitLab CI/CD, CircleCI, and Azure DevOps are tools that implement CI/CD practices.

---

### 2. Automation Reduces Human Error

Automated testing and deployment eliminate many problems caused by manual processes.

---

### 3. A Failed Pipeline Is Helpful

A pipeline failure prevents faulty code from reaching users and highlights issues early in the development lifecycle.
